React는 종종 프레임워크와 라이브러리 사이에서 혼동을 일으키곤 한다. 많은 사람들이 React를 프레임워크로 오해하는 경우가 많지만, 실제로는 **라이브러리**로 분류된다. 이번 글에서는 React가 왜 라이브러리로 간주되는지, 그리고 그 의미에 대해 알아보자.

## TL;DR

- React는 UI 구성에 집중하는 라이브러리로서, 개발자가 필요할 때 호출하여 사용하는 도구이다.
- 반면, Next.js와 같은 프레임워크는 애플리케이션의 전체 구조와 흐름을 정의하고 제어한다.
- React가 라이브러리로 설계된 이유는 UI에 집중하고, 기존 프로젝트에 점진적으로 도입할 수 있으며, 커뮤니티와 생태계 확장을 용이하게 하기 위함이다.

## 라이브러리와 프레임워크

소프트웨어 개발에서 라이브러리와 프레임워크는 모두 코드의 재사용성을 높이고 개발 생산성을 향상시키기 위한 도구이다. 그러나 이 둘은 몇 가지 중요한 차이점이 있다.

### 1. 라이브러리

라이브러리는 특정 기능을 수행하는 코드의 집합체로, 개발자가 필요할 때 호출하여 사용할 수 있다. 라이브러리는 주로 특정 작업을 쉽게 처리하기 위한 도구를 제공하며, 개발자가 애플리케이션의 흐름을 제어한다.

- 장점: 유연성, 선택적 사용, 특정 기능에 집중
- 단점: 일관된 구조 부족, 추가 설정 필요
- 예시: Lodash, Axios, React

### 2. 프레임워크

프레임워크는 애플리케이션의 전체 구조를 정의하고, 개발자가 그 구조에 맞춰 코드를 작성하도록 요구한다. 프레임워크는 [제어의 역전(Inversion of Control)](https://en.wikipedia.org/wiki/Inversion_of_control) 원칙을 따르며, 개발자가 아닌 프레임워크가 애플리케이션의 흐름을 제어한다.

- 장점: 일관된 구조, 내장 기능, 빠른 개발
- 단점: 유연성 부족, 러닝 커브, 특정 패턴 강제
- 예시: Angular, Vue.js, Next.js

## React는 왜 라이브러리로 간주될까

React를 사용할 때 우리는 React의 구조와 문법 등에 맞춰 코드를 작성하도록 요구받는다. 그런데 왜 React는 프레임워크가 아닌 라이브러리로 분류될까?
대표적인 프레임워크인 Next.js와 비교해보면 그 차이를 쉽게 알 수 있다.

터미널에서 아래 명령어를 통해 Vite로 React 프로젝트를 생성해보자.

```bash
npm create vite@latest
```

그리고 `main.tsx` 파일을 열어보면 다음과 같은 코드가 있다.

```jsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import './index.css'
import App from './App.tsx'

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>,
)
```

이 코드는 React 애플리케이션의 진입점(entry point)이다. 여기서 우리는 `createRoot`와 `render` 메서드를 사용하여 React 컴포넌트를 DOM에 렌더링한다. 이 과정에서 우리는 React의 특정 구조에 맞춰 코드를 작성하고 있지만, 애플리케이션의 전체 흐름을 제어하는 것은 아니다. 즉, React는 우리가 필요할 때 호출하여 사용하는 도구(라이브러리)인 것이다.

이번에는 터미널에 아래 명령어를 입력해 Next.js로 프로젝트를 생성해보자.

```bash
npx create-next-app@latest
```

코드 어디를 봐도 `createRoot`나 `render` 같은 React 메서드는 보이지 않는다.

Next.js는 React를 기반으로 하지만, 애플리케이션의 전체 구조와 흐름을 내부적으로 정의하고 제어한다. 예를 들어, 폴더와 파일 기반의 페이지 라우팅, 데이터 페칭, 서버 사이드 렌더링(SSR) 등의 기능이 프레임워크에 내장되어 있다. 개발자는 Next.js가 제공하는 구조에 맞춰 코드를 작성해야 하며, 애플리케이션의 흐름은 Next.js가 제어한다.

즉, Next.js는 프레임워크로서 애플리케이션의 전체 구조와 흐름을 정의하고 제어하는 반면, React는 특정 기능을 수행하는 도구(라이브러리)로서 개발자가 필요할 때 호출하여 사용하는 것이다.

## 왜 React를 라이브러리로 만들었을까

### 1. UI를 구성하는 View만 집중하기 위해

- React는 **UI**를 선언적(declarative)으로 구성하여 Readable한 코드를 작성하고자 하는 목적에서 개발됐다.
- 페이스북 초기 팀에서는 “웹 전체 프레임워크를 만들기보다는 UI 구성 문제를 먼저 해결하자”는 접근을 했다.
- 그래서 데이터 관리, 라우팅, 서버 통신 같은 부분은 React 핵심과 분리되어 있다.
- React가 모든 걸 포함한 프레임워크가 되면, 오히려 복잡성이 증가하고, 선택권이 줄어들 수 있다.

### 2. 기존 프로젝트에 점진적 도입을 가능하게 하기 위해

- 라이브러리의 성격을 유지함으로써 React가 아닌 기존 프로젝트에도 React를 부분적으로 적용할 수 있다.
  - 예: 페이지 일부의 UI만 React로 바꿀 수 있고, 전체 프로젝트는 재작성할 필요가 없음.
- 프레임워크였다면 전체 구조를 React 스타일로 바꾸는 강제성이 있었을 것. (예: Next.js)

### 3. 커뮤니티와 생태계 확장을 용이하게 하기 위해

- React는 **UI 라이브러리**라는 최소한의 정의를 유지하면서, 라우팅이나 상태 관리 등은 외부 라이브러리(React Router, Redux 등)에 맡김.
- 따라서 커뮤니티에서 자유롭게 다양한 솔루션을 만들 수 있도록 했다.

## 결론

React는 UI 구성에 집중하는 라이브러리로서, 개발자가 필요할 때 호출하여 사용하는 도구이다. 반면, Next.js와 같은 프레임워크는 애플리케이션의 전체 구조와 흐름을 정의하고 제어한다. React가 라이브러리로 설계된 이유는 UI에 집중하고, 기존 프로젝트에 점진적으로 도입할 수 있으며, 커뮤니티와 생태계 확장을 용이하게 하기 위함이다.

누군가는 React를 프레임워크라고 부르기도 하지만, 엄밀히 말하면 React는 라이브러리이다. 그러나 React가 라이브러리냐 프레임워크냐를 따지기보다, React가 제공하는 유연성과 강력한 기능들을 활용하여 유저에게 좋은 경험을 전달할 수 있는 애플리케이션을 개발하는데 집중하는 것이 더 큰 의미가 있는 것 같다.
